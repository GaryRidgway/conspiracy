Infinite Whiteboard
Application

A single-page spatial canvas with embedded iframes, connected cards, and cloud-backed persistence. This plan sequences the build so each phase produces something real to react to — then layers in a sparse, low-cost sync model so boards follow you across devices.

Reference

Storage & Sync Architecture

— your data, in your Google Drive

The app never holds your data — there's no server. Each board is a single JSON file in the user's own Google Drive, mirrored to a local cache so the app works offline. The app keeps only a short-lived OAuth token to read/write that one file on your behalf. Delete the app tomorrow and your boards are still in your Drive, openable as plain JSON. Because boards are rarely co-edited and a little lag is fine, sync is just polling the file.

Your file, your Drive store

One JSON file per board, in the user's Google Drive. Accessed via the narrow drive.file scope — the app can only touch files it created or that you explicitly open through Google's picker, never the rest of your Drive. No app database, no data on our side.

Cards as a keyed map model

Store nodes as cards: { id: {…} }, never as an array. Keyed maps make local merges and targeted updates clean, and keep the document diff-friendly.

Write on commit writes

Update the screen locally during a drag; write the file once on release. Debounce text edits to one write a couple seconds after typing stops.

Drive has no field-level patch, so each save rewrites the whole file and bumps version. Two simultaneous saves resolve last-write-wins — the laggy, small-group case you've accepted.

Version-sentinel polling reads

No live listeners. Poll the file's modifiedTime (or the version field) on a timer and on window focus. If it hasn't moved, do nothing; only a change triggers a real download.

Idle sync is a tiny metadata check — comfortably inside Drive's free quotas.

// One board file — lives in the user's Google Drive, not on our servers
{
  "version": 47,            // bumped on every commit; the poll sentinel
  "viewport": { "x": -1200, "y": 340, "zoom": 1.0 },
  "cards": {                  // a MAP keyed by id — not an array
    "c_a1": { "x": 120, "y": 80, "title": "Idea", "body": "…" },
    "c_b2": { "x": 480, "y": 210, "title": "Notes", "body": "…" }
  },
  "iframes": {
    "f_x9": { "x": 900, "y": 60, "w": 640, "h": 400, "src": "https://…" }
  },
  "connections": {
    "cn_1": { "from": "c_a1", "to": "f_x9" }
  }
}
What this buys — and the trade-offs. No backend to run, no per-user database, no data on our side: just OAuth + the Drive API straight from the browser, and sharing is Google's own (share the file view/edit and collaborators open it in the app). In exchange: the app must be served from a registered https origin (localhost for dev) behind a one-time Google Cloud OAuth client; whole-file writes mean two simultaneous saves are last-write-wins; and the narrow drive.file scope keeps setup light by avoiding Google's heavy restricted-scope verification. Non-Chromium quirks don't apply here — any modern browser can do OAuth + Drive; the local-file fallback (export/import) remains for fully offline use.
Track B

Your-Drive Sync & Sharing

— local-first, you own the file
30%
Connect a board to your Drive
Save/load a board as a JSON file in the user's own Google Drive. localStorage stays as the offline cache; the data model doesn't change.
  • Google Cloud OAuth client + Drive API enabled (one-time, by the deployer); served from a registered https origin App ships with a config placeholder for the client ID + a short setup guide
  • "Sign in with Google" via Google Identity Services; narrow drive.file scope only
  • Create a new board file, or open an existing/shared one via the Google Picker
  • Write-on-commit: debounced whole-file save, version bumped each change
60%
Cross-device & shared sync
The same file, current on every device you open it from — and for the small group you've shared it with.
  • User-chosen storage per board — keep it "On this device" (offline, the default) or "Connect Google Drive"; switchable any time, with the current mode shown in the status bar. Offline never requires sign-in.
  • Poll modifiedTime/version on a timer and on window focus; download only on change
  • Merge remote changes into the local view without losing in-progress edits
  • Last-write-wins on simultaneous saves, with a non-blocking "updated elsewhere" notice; graceful offline / token-expiry handling
90%
Sharing & ownership
Hand a board to someone via Drive — and make it obvious the data is theirs.
  • Share through Drive's own dialog (view vs. edit, enforced by Google — not us)
  • "Make a copy" → duplicate the Drive file so a shared board can be forked
  • "Your data lives in your Drive": open-in-Drive link, show the file, export still available

Further out

Per-node documents
If a board outgrows the 1 MiB limit, split nodes into a subcollection. The keyed-map model makes this a mechanical change.
Live co-editing
If real-time multiplayer ever matters, a CRDT (Yjs) wraps the same model. Deliberately out of scope while "rarely simultaneous" holds.
iframe presets
Saved URL lists per iframe node — quick-switch buttons so one frame acts as a browsable panel.
More node types
Images, code blocks, sticky notes — each extends the same base node schema.
Color-coding cards
A per-card accent/background color for visual grouping — a color field on the card, set from a small swatch picker in the header.
Design pass
Revisit the whole visual language — color scheme, type scale, spacing, card/frame chrome, connection styling, toolbar. Current palette is gold-accented (#FFC629) on slate; treat as provisional until this pass. Best done alongside the other deferred polish here.
Board history
Periodic snapshots keyed alongside the board — undo that survives a refresh.
Themed confirm dialog
Replace the native confirm() on destructive actions — starting with "Clear board" — with an in-theme modal, reusing the iframe-creation modal pattern.
Rich-text cards with inline links ✓ shipped
Card bodies are now sanitized HTML edited with a mini toolbar (bold / italic / list) and inline node links — pick a target node, then ⌘/Ctrl-click the link to jump (via frameNode). The card body field changed from a plain string to HTML.
